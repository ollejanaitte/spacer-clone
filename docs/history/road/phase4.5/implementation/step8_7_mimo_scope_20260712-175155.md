- `semanticParity/` は純粋なライブラリモジュール（`#!/` shebang、`process.argv`、`commander`、`yargs` 使用なし）
- `frontend/package.json` に `"bin"` フィールドなし

- `tsconfig.json` は `noEmit: true` のためTypeScript → JS変換不可

**Step 8.7でのCLI配置方針**（rescope_final:216-231の根拠）:

```
新規: frontend/tsconfig.parity-cli.json  ← 専用tsconfig（emit許可）
新規: frontend/scripts/semanticParityCheck.ts  ← CLI entrypoint
```

既存の `typescript` と `@types/node` にのみ依存し、外部パッケージ追加なし。

### 2.7 Node/browser境界

| レイヤー | 環境 | 根拠 |
|---|---|---|
| `semanticParity/` 全モジュール | Node + Browser 両対応 | `process`/`fs`/`path` を直接使用しない。`types.ts`, `compare.ts` 等は純粋関数 |
| `generatedModelParity.ts` | Node | `fromBridgeProject`, `fromLinerBridge` アダプター経由でNode固有のデータソースに依存 |
| CLI entrypoint (`scripts/semanticParityCheck.ts`) | Node専用 | `node:fs` 使用が必須 |
| `frontend/` 全体 | Browser + Vite dev | `node:crypto` は `src/polyfills/nodeCrypto.ts` にエイリアス設定済み（`vite.config.ts`） |

**CLI実行時のNode/Browser問題**:
- CLI用tsconfig (`tsconfig.parity-cli.json`) は `module: ESNext`, `moduleResolution: Node16` or `Bundler` で生成する必要がある
- ESM出力の場合、`node:` prefix imports はNode.js 18+で自動的に解決可能
- プロジェクトのNode要件は `@types/node: ^25.9.1`（package.json devDependencies）

### 2.8 専用tsconfig構成

**既存tsconfig**:

| ファイル | target | module | moduleResolution | noEmit | include |
|---|---|---|---|---|---|
| `tsconfig.json` | ES2022 | ESNext | Bundler | true | `src` |
| `tsconfig.node.json` | ES2023 | ESNext | Bundler | true | `vite.config.ts` |

**Step 8.7要件**（`tsconfig.parity-cli.json`）:

```

outDir: dist-parity-cli or 類似

```

**制約**: `tsconfig.json` の `include: ["src"]` は `scripts/` を含まないため、CLIスクリプトは別tsconfig管理が必須。

### 2.9 ESM import解決

- `tsconfig.json`: `module: ESNext`, `moduleResolution: Bundler`
- `tsconfig.node.json`: `module: ESNext`, `moduleResolution: Bundler`
- `package.json`: `"type"` フィールド未設定（デフォルトは CommonJS）
- Vite設定: `import.meta.url` を使用（`vite.config.ts:3`）

**CLI ESM配慮**:
- TypeScript emit後、出力JSファイルのimport文は `module: ESNext` により `.js` 拡張子なしのimportパスが出力される可能性がある
- Node.js ESMでは `.js` 拡張子が必須 → tsconfigで `rewriteRelativeImportExtensions` を検討、またはBundler moduleResolutionのままNodeで実行する方法を検討
- `tsx` を使わない方針（rescope_final:229）のため、tsc emit → node実行フローが唯一の選択肢
