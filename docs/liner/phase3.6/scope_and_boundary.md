# Phase 3.6 Scope and Boundary

## 0. 目的

Phase 3.6 JIP-LINER Importer は、JIP-LINER の PDF 出力帳票を見ながら、現場担当者が数値を写経入力するための独立モジュールである。

本フェーズは線形計算そのものを再実装するのではなく、PDF 帳票上に既に出力された計算結果を structured data として保存し、必要に応じて Phase 3.5 vNext draft へ片方向変換する。

## 1. 対象案件例

対象案件例として `第002章_橋梁予備設計_2.2 設計条件_道路条件と設計条件.pdf` に掲載される Hランプ4号橋、およびローカル参照資料 `REPORT09_2編-01_Hランプ4号橋_線形計算書.PDF` を想定する。

Phase 3.6 の写経対象は、`REPORT09_2編-01_Hランプ4号橋_線形計算書.PDF` の 12 ページ目以降に記載される線形入力・計算結果データである。ユーザーはこの PDF を画面または紙で読み、Importer UI に数値を転記する。

扱うデータ範囲:

- 橋梁名、路線・ランプ名、橋梁単位のメタデータ
- 座標系情報（水平、鉛直を分離）
- 径間、支点、ピア、スパンの基本構成
- 基準ラインと橋軸線マスタ
- 横断面ごとの方位角、累加距離、測点
- 橋軸線ごとの X/Y、計画高、横断勾配、単距離、累加距離、単幅、累加幅、交角、測点
- `********` で示される未定義値
- 補助入力としての平面要素、縦断要素、横断勾配定義

## 2. Phase 3.5 との境界

Phase 3.5 はアプリ内で線形計算ロジックを扱う 6 タブ UI である。Phase 3.6 は PDF 帳票を写経する importer UI であり、操作モデル、保存形式、バリデーション、画面遷移を別系統で持つ。

境界ルール:

- Phase 3.5 の 6 タブは改修対象にしない。
- Phase 3.6 画面を Phase 3.5 のタブ内に組み込まない。
- Phase 3.6 の保存 JSON は Phase 3.5 draft とは別ファイルにする。
- Phase 3.6 から Phase 3.5 への接続は adapter による明示エクスポートだけにする。
- Phase 3.5 pipeline / grid / frame は変換後の draft に対して再利用する。

## 3. 非スコープ

以下は Phase 3.6 の非スコープである。

| 項目 | 扱い | 理由 |
|---|---|---|
| JIP-LINER `.LIN` ファイル直接読込 | 非スコープ | PDF 写経入力が本フェーズの目的 |
| LINER 逆エクスポート | 非スコープ | 片方向変換のみ。将来 Phase 3.7 以降 |
| Phase 3.5 -> Phase 3.6 逆変換 | 非スコープ | importer JSON は原票写経の sourceRef を持つため逆生成できない |
| ハンチ計算 | 非スコープ | 構造細部計算であり線形 importer の責務外 |
| 舗装厚計算 | 非スコープ | 設計条件管理の別領域 |
| ループ橋対応 | 非スコープ | LINER 自体が非対応であり本フェーズも扱わない |
| PDF OCR / 表抽出 | 非スコープ | 新規依存導入が必要になりやすく判断が別 Phase |
| 新規 npm パッケージ導入 | 非スコープ | Phase 3.5 と同じ制約 |
| Feature Flag | 非スコープ | 独立画面としてブランチ運用で切り分ける |

## 4. 想定する橋梁形態

対応対象:

- 単純橋
- 連続橋
- 上下線一体橋
- 上下線分離橋
- 複数径間を持つ一般的なランプ橋
- 径間内で橋軸線セットが切り替わる案件

制限:

- 1 つの横断面は 1 つの PDF ページに対応することを前提にする。
- 1 ページ内の行列構造が大きく異なる帳票は、Phase 3.6 初版では手動補正または対象外診断とする。
- 複数橋梁を 1 importer project に含めるかは M7 の確認事項とする。

## 5. 成果物境界

本設計パックで作成するのは M0〜M7 の Markdown 設計書だけである。

実装フェーズで作成するもの:

- TypeScript 型
- JSON Schema 実体
- Adapter 実装
- CRUD / ファイル I/O
- UI コンポーネント
- バリデータ
- エクスポート処理
- テストコード
