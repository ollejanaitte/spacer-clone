## 1. リスク

| Risk | 影響 | 対策 |
|---|---|---|
| Phase 3.5 vNext schema が本フェーズ実装中に変わる | adapter の写像変更が必要 | M5 を adapter 契約として維持し、PR-3.6-5 で最終追随 |
| JIP-LINER PDF のフォーマット差異 | 1 ページ = 1 section の UI が合わない可能性 | row/col sourceRef を柔軟にし、手動補正可能にする |
| `LINER入力.pdf` がテキスト抽出できない | 入力 PDF 単体からは機械抽出値で埋められない | ユーザーが読む資料は `REPORT09_2編-01_Hランプ4号橋_線形計算書.PDF` の 12 ページ目以降とし、必要値を転記する |
| 点群のみで線形要素を復元できない | Phase 3.5 pipeline に渡せない | `AlignmentMetadata` 補助入力を必須に近い扱いで warning/error 化 |
| sourceRef が細かく payload を肥大化させる | JSON が重くなる | conversion log と payload の責務分離を検討 |
| 現場担当者に座標系入力が難しい | 入力完了率が下がる | 不明入力を許容し、export warning にする |
| 既存 UI と importer UI の導線が混同される | Phase 3.5 との境界が曖昧になる | 起動導線、画面タイトル、保存形式を分離 |
| 新規 npm パッケージ禁止により PDF 貼り付け支援が限定的 | 半自動化が弱い | Phase 3.6 は手入力を主対象とし、OCR は将来判断 |

## 3. エスカレーション条件

以下が発生したら作業を停止して確認する。

- Phase 3.5 vNext schema と Phase 3.6 data model が M5 の写像で吸収不能に衝突する。
- 既存 6 タブと共存する UI 導線が既存アーキテクチャで表現不能。
- JIP-LINER PDF の実データが「1 ページ = 1 横断面、15 行 × 9 列程度」という想定と大きく異なる。
- ワイヤーフレームで表現しきれない UX 判断が必要になる。
- PDF OCR 導入や複数橋梁同時編集など、Phase 3.6 の前提を変える判断が必要になる。
