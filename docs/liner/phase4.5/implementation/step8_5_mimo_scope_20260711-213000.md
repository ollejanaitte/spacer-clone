## 2. 現行LoadCase型

### 2.1 `frontend/src/types.ts` (L59–63)

```typescript
export type LoadCase = {
  id: string;
  name: string;
  type: "static";
};
```

**分析:**
- `type` は `"static"` のみ。`"eigen"`, `"response_spectrum"` は `analysisSettings` に記述されるが `LoadCase.type` としては未使用。
- `id` と `name` は別個。legacy path は `id` を自動採番、BD path は `caseId` を `BridgeDefinitionLoad.caseId` から転記。
- **semantic identity 規則:** `(name, type)` を primary key とし、`id` は trace metadata とする。理由: generator path が異なれば `id` は一致しないが、設計意図（荷重ケース名＋種別）は一致する。

### 2.2 `backend/engine/model.py` (L86–90)

```python
@dataclass(frozen=True)
class LoadCase:
    id: str
    name: str
    type: str
```

**分析:**
- Python 側は `type: str` で型制限なし。frontend と整合。
- `validate_model` で `ensure_unique` により `id` のユニーク性を検証。

### 2.3 generator (`structuralModelGenerator.ts` L555–558)

```typescript
const ensureLoadCase = (caseId: string, name?: string) => {
  if (!loadCases.some((loadCase) => loadCase.id === caseId)) {
    loadCases.push({ id: caseId, name: name ?? caseId, type: "static" });
  }
};
```

**分析:**
- BridgeDefinition の `load.caseId` から `LoadCase` を生成。
- `name` は `load.id` を使用（設計荷重ID）。
- placeholder: `loadCases.length === 0` 時に `PLACEHOLDER_LOAD_CASE_ID` を追加。
- **legacy path の `LoadCase` 生成は `bridge_fem_generator.py` に依存**（Python backend）。

## 6. Load Case Identity

### 6.1 Semantic Key 規則

```
loadCaseSemanticKey = `${name}:${type}`
```

- `id` は trace metadata として記録するが、比較キーには含めない。
- **理由:** legacy path は `id` を自動採番（例: `"LC1"`）、BD path は `BridgeDefinitionLoad.caseId` から転記（例: `"LC1"`）。設計意図が同一でも `id` が異なる場合がある。
- **`name` が同一で `type` が同一** → 同一 load case と判定。

### 6.2 Generator での LoadCase 生成

| 経路 | LoadCase.id | LoadCase.name | LoadCase.type |
| --- | --- | --- | --- |
| Legacy Python | `"LC1"` 等（自動採番） | `"自重"`, `"活荷重"` 等 | `"static"` |
| BridgeDefinition | `BridgeDefinitionLoad.caseId` | `BridgeDefinitionLoad.id` | `"static"` |
| LINER | 空配列（loads 未マッピング） | — | — |

### 6.3 Fixture での検証

- `self_weight` case: `name="self_weight"`, `type="static"` → 両経路同一前提。
- `distributed` case: `name="distributed"`, `type="static"` → 両経路同一前提。

## 7. Nodal Load Identity

### 7.1 Semantic Key 規則

```
nodalLoadSemanticKey = `${nodeKey}:${loadCaseSemanticKey}:${fx}:${fy}:${fz}:${mx}:${my}:${mz}`
```

- `nodeKey` は Step 8.2 で確立済みの座標ベース key（`normalize.ts`）。
- **6 DOF 全コンポーネントが同一** → 同一 nodal load と判定。
- `id` は trace metadata。

### 7.2 Matching 手順

1. 既にマッチ済みの `nodeMatch.matched` を使用。
2. 各マッチペアについて、left と right の `nodalLoads` を `loadCaseSemanticKey` でグループ化。
3. 同一 `loadCaseSemanticKey` 内で、6 DOF コンポーネントの全一致を確認。
4. 全一致 → matched。不一致 → mismatch（severity: blocker）。
5. 片側のみに存在 → unmatched（severity: error）。

## 8. Member Load Identity

### 8.1 Semantic Key 規則

```
memberLoadSemanticKey = `${endpointKey}:${loadCaseSemanticKey}:${coordinateSystem}:${wx}:${wy}:${wz}`
```

- `endpointKey` は Step 8.2 で確立済みの canonical endpoint key（`normalize.ts` L38–40）。
- **`coordinateSystem` が semantic key の主要要素。** `"local"` と `"global"` は意味が異なる。
- `memberId` は trace metadata。

### 8.2 I/J 反転の扱い

- Step 8.2 の `canonicalEndpointKey` は `[nodeIKey, nodeJKey].sort()` により I/J 反転を正規化。
- **MemberLoad は endpoint key でマッチングされるため、I/J 反転 member 上の load も同一と判定。**
- **ただし `coordinateSystem: "local"` の場合、I/J 反転により局所座標系が反転する可能性がある。**
  - **判定:** `coordinateSystem: "local"` の MemberLoad は、endpoint key が同一でも局所軸方向の違いにより結果が異なる場合がある。
  - **対応:** Step 8.5 では `coordinateSystem: "local"` の MemberLoad について、局所座標変換を考慮した比較を実装するか、diagnostic warning を出力する。
  - **現行 generator は `coordinateSystem: "global"` のみ使用**（`structuralModelGenerator.ts` L649）。legacy path のみ `"local"` を使用する可能性がある。

### 8.3 Matching 手順

1. 既にマッチ済みの `memberMatch.matched` を使用。
2. 各マッチペアについて、left と right の `memberLoads` を `loadCaseSemanticKey` でグループ化。
3. 同一 `loadCaseSemanticKey` + `coordinateSystem` 内で、`wx`, `wy`, `wz` の全一致を確認。
4. 全一致 → matched。不一致 → mismatch（severity: blocker）。
5. 片側のみに存在 → unmatched（severity: error）。

## 9. Coordinate System (Local/Global)

### 9.1 現行実装

| 経路 | coordinateSystem | 備考 |
| --- | --- | --- |
| Legacy Python | `"local"` or `"global"` | solver で局所→グローバル変換 |
| BridgeDefinition generator | `"global"` のみ | `structuralModelGenerator.ts` L649 |
| LINER | 未マッピング | `fromLinerBridge.ts` は `loads: []` |

### 9.2 Semantic Parity での扱い

- `coordinateSystem` は MemberLoad の semantic key に含める。
- `"local"` vs `"global"` は **意味が異なる**ため、同一 member 上の同一荷重でも `coordinateSystem` が異なれば **mismatch** と判定。
- **Step 8.5 のスコープ内:** `"local"` と `"global"` の比較は、endpoint key + coordinateSystem + load components の全一致で判定。
- **Step 8.6 のスコープ外:** 解析結果（member end force）の I/J 反転変換は Step 8.6 で扱う。

## 10. Explicit Zero と Absent の区別

### 10.1 定義

| 状態 | 定義 | Parity 判定 |
| --- | --- | --- |
| **Absent** | 一方のモデルにその load が存在しない | unmatched → severity: error |
| **Explicit Zero** | load が存在し、全コンポーネントが 0.0 | matched → equivalent（zero load） |
| **Non-zero** | load が存在し、少なくとも1コンポーネントが非零 | matched → 比較 |

### 10.2 Implementation

- `normalizeProjectModelForSemanticParity` は現在 `nodalLoads`, `memberLoads` を正規化していない。
- **Step 8.5 で追加:** `NormalizedModel` に `loadCases`, `nodalLoads`, `memberLoads` を追加。
- **Absent vs Zero の検出:**
  - left に load がなく right に存在 → `unmatchedRight` に追加。
  - 両方に存在し全コンポーネント 0 → matched（zero load）。
  - 両方に存在し不一致 → mismatch。

### 10.3 Legacy vs BD の self-weight 差異

| 項目 | Legacy | BD |
| --- | --- | --- |
| self-weight | `NodalLoad.fz` を等分配 | `NodalLoad.fz` を等分配 |
| load case ID | 自動採番 | `BridgeDefinitionLoad.caseId` |
| 総荷重 | Σ(fz) = 設計値 | Σ(fz) = 設計値 |

- **設計意図が同一** → total applied load は一致するはず（M016）。
- **ノード数が異なる** → 分配荷重の個別値は異なるが、**総荷重の一致**が BLOCKER 条件。

## 11. Duplicate Load の扱い

### 11.1 定義

同一 semantic key（endpointKey + loadCaseSemanticKey + coordinateSystem + components）を持つ複数の load。

### 11.2 判定規則

- **同一モデル内の duplicate:** `validate_model` で `ensure_unique` により `id` のユニーク性を検証。semantic duplicate は許容（例: 同一 member 上の同一 case の複数 uniform load → solver で加算）。
- **cross-model duplicate:**
  - left に2件、right に1件 → **ambiguity**（`indeterminate`）。
  - left に2件、right に2件 → pair-wise matching を試行。全ペア一致 → matched。不一致あり → ambiguity。
  - **Step 8.5 の実装方針:** duplicate candidate は `AmbiguousMatch` として記録し、`indeterminate` に昇格。

### 11.3 Assembly での加算

`assembly.py` L95–110: 同一 memberId + loadCaseId の複数 MemberLoad は加算される。
- **Semantic parity では:** 各 load を個別に比較し、加算結果は Step 8.6 の解析結果比較で検証。

## 14. Node/Member ID 非依存

### 14.1 規則

- `nodeId`, `memberId`, `loadCaseId` は比較キーに使用しない。
- **対応:** 既にマッチ済みの `nodeMatch.matched`, `memberMatch.matched` を使用し、semantic key ベースで load を関連付ける。

### 14.2 実装

```
left nodalLoad.nodeId → left node.sourceId → left node.key → matched node pair → right node.key → right nodalLoads (same loadCaseSemanticKey)
```

- `nodeId` → `nodeKey` の変換は `normalizeNodes` で完了済み。
- `memberId` → `endpointKey` の変換は `normalizeMembers` で完了済み。

## 22. 比較関数設計

### 22.1 `loadParity.ts` の主要関数

```typescript
export function normalizeLoadCases(
  loadCases: LoadCase[],
  diagnostics: { warnings: SemanticParityDiagnostic[]; errors: SemanticParityDiagnostic[] },
): NormalizedLoadCase[];

export function normalizeNodalLoads(
  nodalLoads: NodalLoad[],
  nodesById: Map<string, NormalizedNode>,
  loadCasesBySemanticKey: Map<string, NormalizedLoadCase>,
  diagnostics: { warnings: SemanticParityDiagnostic[]; errors: SemanticParityDiagnostic[] },
): NormalizedNodalLoad[];

export function normalizeMemberLoads(
  memberLoads: MemberLoad[],
  membersByEndpointKey: Map<string, NormalizedMember>,
  loadCasesBySemanticKey: Map<string, NormalizedLoadCase>,
  diagnostics: { warnings: SemanticParityDiagnostic[]; errors: SemanticParityDiagnostic[] },
): NormalizedMemberLoad[];

export function compareLoadParity(
  leftModel: NormalizedModel,
  rightModel: NormalizedModel,
  nodeMatches: MatchedPair[],
  memberMatches: MatchedPair[],
  tolerance: SemanticTolerance,
): {
  summary: LoadParitySummary;
  mismatches: ParityMismatch[];
  ambiguities: AmbiguousMatch[];
};
```

### 22.2 `compareLoadParity` のアルゴリズム

1. **Load Case Set 比較:**
   - left/right の `loadCases` を `name:type` で集合比較。
   - 不一致 → mismatch（severity: error）。

2. **Nodal Load 比較:**
   - `nodeMatches` を使用してマッチしたノード上で比較。
   - 各マッチペアについて、`loadCaseSemanticKey` でグループ化。
   - 同一 `loadCaseSemanticKey` 内で、6 DOF コンポーネントの全一致を確認。
   - **total applied load:** 各 load case について Σ|F| を比較（relTol 1e-6）。

3. **Member Load 比較:**
   - `memberMatches` を使用してマッチした member 上で比較。
   - 各マッチペアについて、`loadCaseSemanticKey` + `coordinateSystem` でグループ化。
   - 同一グループ内で、`wx`, `wy`, `wz` の全一致を確認。

4. **Ambiguity 処理:**
   - duplicate candidate は `AmbiguousMatch` として記録。
   - `ambiguousLoadCaseCount > 0` → status は `indeterminate`。

5. **Status 判定:**
   - `load.caseSetEquivalent === false` → `different`。
   - `totalLoadMismatchCount > 0` → `different`。
   - `ambiguousLoadCaseCount > 0` → `indeterminate`。
   - 全て pass → `equivalent`（他の metric と組み合わせて最終 status を決定）。

## 26. Acceptance Criteria

### 26.1 必須条件

1. **Load case set equality:** ordering と non-semantic ID 差異を無視して、load case 集合が一致すること。
2. **Nodal load matching:** semantic node key 経由で一致すること。
3. **Member load matching:** semantic member endpoint key + coordinate basis 経由で一致すること。
4. **Total applied load:** 各 load case について Σ|F| が relTol 1e-6 以内であること（M016）。
5. **Absent vs zero:** absent item は explicit zero item と異なると判定されること。
6. **Unsupported domain:** 未対応の荷重/境界条件は not implemented として報告され、silent equivalent とならないこと。
7. **Existing regression:** 既存 support/property parity テストに影響がないこと。
8. **Typecheck:** `npm run typecheck` が通ること。

### 26.2 M015–M017 Style Metrics

| Metric | Step 8.1 ID | 比較方法 | 許容差 | Severity |
| --- | --- | --- | --- | --- |
| Load case set equivalence | M015 | exact set match | — | ERROR |
| Total applied load per case | M016 | Σ\|F\| or ΣF_z | relTol 1e-6 | BLOCKER |
| Load direction unit vector | M017 | 代表方向 | absTol 1e-6 | ERROR |
