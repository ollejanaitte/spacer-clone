# Language Policy

This project uses two distinct languages for two distinct audiences:

1. **Internal implementation language: English.** All source code identifiers (variables, types, functions, hooks, components, modules), all code comments, and all design documents under `docs/` are written in English. The recommended English wording for Japanese domain terms is recorded in `docs/glossary.md`.

2. **User-facing language: Japanese.** All visible UI text — button labels, menu items, dialog text, validation messages, report labels — is Japanese. UI strings are stored in a single localization module, `frontend/src/i18n/ja.ts`, and are imported by components. Inline Japanese strings inside JSX or component code are not allowed.

## Scope

| Area | Language | Notes |
| --- | --- | --- |
| Source code identifiers | English | Required. |
| Code comments | English | Required. |
| Design documents in `docs/` | English | Required. |
| API request/response JSON keys | English | Required. Persisted project / result file keys are part of the schema and are not renamed lightly. |
| Persisted project / result fields | English | Existing keys are kept stable for backward compatibility. |
| User-visible UI strings | Japanese | Required. Source of truth is `frontend/src/i18n/ja.ts`. |
| User-visible report labels | Japanese | Required. Generated by the backend report module from Japanese label tables. |
| Identifier-style Japanese (e.g. `節点` as a function name) | Forbidden | Use the English term from the glossary. |

## What is intentionally not changed

To keep this refactor strictly behavior-preserving, the following are out of scope:

- **JSON schema keys** for `project.json`, `bridge.schema.json`, `generated-fem.schema.json`, `result.schema.json`, and any saved project or result files. These are part of the on-disk format and are not renamed.
- **HTTP request/response field names** for the FastAPI endpoints. The API contract is stable.
- **Numerical constants, formulas, analysis result structures, and computation logic.** The Japanese-language refactor touches identifiers and comments only.
- **User-facing UI strings.** They stay in Japanese; this refactor only relocates them into `ja.ts`.
- **`backend/data/projects/autosave.json`.** It is in `.gitignore` and is regenerated at runtime; it is left untouched.
- **Generated build artifacts** under `frontend/dist/`, `desktop/electron/dist/`, `release/`, `build/`, and `dist/`. They are rebuilt by the build pipeline.
- **Sample JSON files** under `examples/`. Their schema-key fields remain unchanged.

## Where Japanese strings are centralized

UI text is centralized in:

- `frontend/src/i18n/ja.ts` — the single source of truth for visible Japanese labels, grouped by feature/component (toolbar, results panel, bridge wizard, viewer, property panel, dialogs, etc.).

When adding a new visible label:

1. Add the Japanese string to `frontend/src/i18n/ja.ts` under an appropriate group.
2. Import the constant in the component file.
3. Render it in JSX.
4. Do not write Japanese strings inline in JSX or in `setMessage(...)` style helpers; always use a constant from `ja.ts`.

## Adding a new domain term

When a new Japanese term is needed:

1. Add the Japanese / English pair to `docs/glossary.md`.
2. Use the English word in code identifiers and design documents.
3. Use the Japanese word only as a UI string in `ja.ts`.
